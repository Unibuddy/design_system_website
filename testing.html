Testing

Overview

Testing the front end components is currently done in two general ways, shallow and mount tests both using enzyme provided functions (react-native only supports shallow), and a newer way using react-testing-library from Kent C. Dodds, and react-testing-library is generally preferred.

The react-testing-library encourages testing closer to what the user does in the browser by providing easy selectors from box and disables accessing the component internal state and props. In fact this library "pretends" to not know react even exists and acts as an extension of dom-testing-library. Besides a plethora of useful selectors another way it helps with integration tests is by providing "wait" utilities.

We are also using the jest dom library for better dom assertions.

Problem

When writing integration tests the biggest issue usually is async nature of the browser interaction and React rendering cycles. For example we will usually have the following test scenario:

import {render, fireEvent} from "react-testing-library";
test("should change after click", () => {
    const {getByTestId} = render(<MyComponent />);
    fireEvent.click(getByTestId("some-button"));
    expect(getByTestId("some-dynamic-element")).toBeInTheDocument();
});

This will often fail, because the rendering will be async for any number of reasons. When using forms, for example, the validation is async, so any errors and form submission will not be executed immediately. Or if the button triggers a query, and only after it is resolved, the dynamic element will be available.

The way to solve this is to use wait or waitForElement from async utilities

- import {render, fireEvent} from "react-testing-library";
+ import {render, fireEvent, wait, waitForElement} from "react-testing-library";
- test("should change after click", () => {
+ test("should change after click", async () => {
    const {getByTestId} = render(<MyComponent/>);
    fireEvent.click(getByTestId("some-button"))
-   expect(getByTestId("some-dynamic-element")).toBeInTheDocument();
+   await wait(() => expect(getByTestId("some-dynamic-element")).toBeInTheDocument());
    // or...
+   await waitForElement(() => getByTestId("some-dynamic-element"));
});

Prefer mount over shallow

Going forward we should prioritise integration tests for higher level components, and tests that are closer to what the browser renders for the low level ones. Keep in mind that enzyme's mount function uses JSDOM to render the react tree in to a "virtual" browser, and one of the downsides of this is that the elements, for example, have no dimensions. In case you want to test element size, scroll position, visibility etc. this is not possible with JSDOM.

In many cases this will be difficult and/or time consuming since this is a relatively new practice and for high level components it requires a lot of boilerplate. We should aim to create reusable tools, utilities, and boilerplate for testing wherever possible.

Using mount over shallow has one major benefit, and that is that we are not relying on enzyme internals as much as we do when we are using shallow. Since shallow implements its own renderer the enzyme library has to manually add support for every new React feature, and this has been historically very slow to happen.

Integration mocking

In order to mount or render a high level component you might need some or all of the following setup:

  • MockedProvider from react-apollo/test-utils in order to mock queries and mutations
  • Provider wrapper from rect-redux with a new mocked store
  • MemoryRouter wrapper from rect-router if the component is using a withRouter hoc or any of the react router-dom components
  • mocking imported modules using jest.mock()

Apollo mocking

Apollo react testing documentation We should aim to reuse university, user and other common query and mutation mock files, and aggregate in one place.

const mocks = [
    {
        request: {
            query: someMutation,
            variables: {
                thingId: "1",
                newValue: "hello",
            },
        },
        result: {
            loading: false,
            data: {
                someMutation: { thingId: "1", newValue: "hello" },
            },
        },
    },
];
const wrapper = mount(<MockedProvider mocks={mocks} addTypename={false}>
  <SomeComponmentThatUsesApollo/>
</MockedProvider>

Writing tests as documentation in jest

  • use describe blocks to describe the condition
  • use it blocks to test each of the expected results under that condition
describe("when condition x", () => {
  it("should render x", () => {...});
  it("should call prop x with data", () => {...});
  it("should NOT render y", () => {...});
});
describe("when condition y", () => {
  it("should render y", () => {...});
  it("should NOT call prop x", () => {...});
});
