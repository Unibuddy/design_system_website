Contributing

Our main goal with this library is to make building high quality, accessible, and highly polished products in Unibuddy efortless. The purpose of Experimental is to create an extensible library of components that can be easily used across all of our products, and if need be, easily extended in functionality or design for use in a specific app. Furthermore, it is essential that all components are following the WCAG AA accessibility standards and best practices.

What belongs in Experimental

Whenever you are adding a component to Experimental you should think about it in those terms. Let's take the following example. A button that copies text to clipboard. Its behaviour is generic enough, and it is something we need across products. 

import React from "react";
import Tooltip from "@unibuddy/machop/Experimental/General/components/Tooltip/Tooltip";
import Button from "@unibuddy/machop/Experimental/General/components/Button/Button";
import useClipboard from "@unibuddy/machop/Experimental/hooks/useClipboard";
const ButtonCopyToClipboard = ({textToCopy, label, size, icon}) => {
    const Icon = icon;
    const {canUse, copyToClipboard} = useClipboard();
    const onClick = () => {
        copyToClipboard(textToCopy);
    };
    if (canUse) {
        return (
            <Tooltip label={label}>
                <Button iconOnly size={size} onClick={onClick} data-test-id="clipboardButton">
                    <Icon aria-hidden />
                </Button>
            </Tooltip>
        );
    }
    return null;
};
export default ButtonCopyToClipboard;

Can this component be used in the widget? How about live events? We cannot be sure, because the design requirements will be different depending on the context. This button has a very specific look, icon only button with a specific color. It has a restrictive api, we can only pass the icon and the size, and it is therefore not extensible. It is also not composable, we cannot pass a different button, or altogether different component to it.

The reason api design is paramount when it comes to making design system components is that its main purpose should be speed of development. If it takes longer to customize the component for our need than to make one from scratch, it actually hinders our development process rather than helps. Unless the component is not accessible, imports a large library, or it has serious performance issues, the implementation is not important. We can fix that later. But fixing a bad legacy api is an order of magnitude more difficult. This is the case with most abstractions.

That is why it is often better to leave the rendering to the app if we are not sure of the cross-product requirements and design. In this case, the useClipboard hook from Experimental is the important part and the way the button looks can easily be re-made on a case by case basis.

Going forward we should implement a strategy of separating components. Components that satisfy any of these rules belong in Machop: 1. If the component is a template or a page 2. if the component is a collection of other DS components 3. if the component uses a specific apollo query or mutation 4. If the component implements product(s) specific business logic

When adding components to Experimental keep in mind:

  • This component should be reusable across products
  • Therefore, colours, sizing and other design factors should be customisable, theme-able and work across different contexts
  • It should be extensible through composition, see how SubmitButton composes over LoadingButton, and LoadingButton composes over Button. LoadingButton passes all the props down, only adds loading functionality, but also is aware of how Button is styled
  • component should be well documented

Requirements

  • yarn
  • node v10+

Pre-commit hooks

For easier development and git diffing we run yarn lint and prettier as a pre-commit hook. If your commit fails check the output, it is likely that one of these two tasks failed.

Overview

Dependency management

We use yarn workspaces to manage our front end client's dependencies. You can install all of the dependencies by running yarn from the the root clients folder or any of the "packages". It is not necessary to run yarn command in each of the sub-folders.

Code sharing between apps

Experimental should not import any dependency outside itself.

Code style

All code style is enforced by prettier, which you should install in your editor, and then you can for the most part not worry about it. However, in order to enforce consistency, we have a pre-commit automatic prettier run on files that are staged for that commit. This way you should never have code style clashes in you PR's.

Linting

Code quality is enforced through eslint, and you should have it installed in your editor (and globally in order to work in the editor). yarn lint should be run before opening a PR in order to check if all the linting is passing. yarn lint:fix can be run to automatically fix some of the linting issues. Eslint should not be concerned with any code styling, only catching common js and React problems.

Javascript version

We write our javascript code in ES2018+ style with the following TC39 proposals that are allowed:

  • class fields
  • static class features
  • import()

Adding any new experimental js features or proposals needs to be discussed with the rest of the team. Ideally the proposal should be at least past stage 2 at the TC39 committee, meaning it is pretty much guaranteed to end up in browsers.

Common dependencies

We are using the latest React version 16.8, and the following libraries:

  • apollo-client
  • graphql
  • react-router
  • styled-components
