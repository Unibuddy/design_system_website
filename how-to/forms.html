Forms

The main concept we need to get up to speed on regarding the form handling in react is the controlled vs uncontrolled forms. Relevant react documentation Forms.

Controlled vs. Uncontrolled

By default, input elements in HTML and in react are uncontrolled, meaning they maintain internal state. React documentation, however, recommends that you build controlled components in most cases, meaning you listen to onChange event, set the state from that event value, and then pass that state to the input element value prop. It turns out, this is extremely tedious and too low level when building products, and makes building large forms very slow and bug prone.

This is the reason there are so many form libraries out there, and most of them focus on two things, maintaining internal control of the state, and validation, while offering some convinient api's as escape hatches. We decided to use Formik for managing form state, and yup for validation.

Get Started

To start with using our form components you must wrap your inputs with the Form components, add a SubmitButton along the inputs, and finally add an onSubmit prop to the Form. You can submit the Form by either pressing enter when focused on one of the text inputs, or by pressing the SubmitButton. The onSubmit callback will get called with two arguments - first is the form values, and the second is an object which gives you the imperative methods to call to finish submitting, set an error on the form, or manipulate the internal Form state in some other way.

In most cases you will want to add the initialValues and validationSchema which are both plain object props. If the Form fields don't pass validation step, the onSubmit will not get called. As for the initialValues, besides initializing the form state, it is also useful to let the Form know that a certain field exists in the form. This is useful when showing errors after the first attempt at submitting. If the form doesn't know that the field exists, it will not show the error for that field. The other way the form will know a field exists is when the user interact with the field, but this may not happen.

Migrate from a controlled form

Our form components are a thin wrapper around Formik. To start wrap all your form inputs in a <Form> component.

Next we need to wrap the input in a <FormGroup> component in order to wire up the input with the <Form>. Next we can remove the value and onChange props as we don't need to control Formik components, the <Form> holds all the state. Our page only declares what inputs it needs.

We add label="Test" prop that internally adds a label element and associates it with the input via a generated or manually provided id. Why we do this.

Finally, we need a button with a type="submit", in order to be able to submit the form by either clicking on it or via enter press.

In order for formik to differentiate inputs we need to add a name to each of them. See name="test" First, we will replace input with TextInput for some nice default styling. We can replace also the button with a <SubmitButton> that abstracts away the disabling of the button and loading state.

Adding validation

Finally to validate the form we add validationSchema and initialValues. Initial values is important because that way formik knows which fields exist in the form and therefore are valid/invalid before any onChange is fired.

onSubmit

In order to get the data from the form we must add a onSubmit callback to the Form component. On form submit Formik asynchronously validates the form and if all the validation passes calls our onSubmit callback with values as the first argument and internal actions as the second arg.

In order for the serverError error message to show we have to add it to the initial values, otherwise the form will not know that that it exists and formik never shows errors on fields that haven't been interacted with.

Testing forms

In the previous section we mentioned that the form validation is async, which means you will need to wait for changes in the form in your tests. The testing library of choice is @testing-library/react because it provides easy async testing utilities like waitFor and findBy*. RTL async utilities docs. Notion doc for testing forms

Adding a custom form input

Never add a new form input that cannot be used outside of Form context!

In order to add a new input type that interfaces with <FormGroup and <Form> it needs to implement a kind of interface. The input needs to be controlled, meaning it has to accept a value and name props and emit props.onChange and props.onBlur that emit a "react like" event.
